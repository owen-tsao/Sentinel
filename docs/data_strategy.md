# Data Strategy

## Short Answer

You should not rely only on hand-written examples, but you should keep a small hand-labeled seed set.

The efficient path is:

1. Use `data/examples/starter_seed.jsonl` as the label policy and sanity-check set.
2. Pull larger examples from public agent-safety benchmarks.
3. Use GPT-generated examples to fill targeted gaps.
4. Validate and sample-check generated labels before training.

## Why Not Use Only Online Data?

Public benchmarks are useful, but they may not map perfectly to Sentinel's exact API shape. Some are task-level, GUI-heavy, or do not provide clean `(context, command)` pairs. A small seed set defines what Sentinel means by `safe`, `authorized_destructive`, `confirm_required`, and `malicious`.

## Why Not Use Only GPT Data?

GPT can generate many examples quickly, but it can also:

- overrepresent obvious attacks,
- label destructive-but-authorized commands incorrectly,
- miss realistic benign developer workflows,
- generate commands that are syntactically odd or unrealistic,
- leak your own assumptions back into the dataset.

Use GPT to expand coverage, not to define the ground truth alone.

## Recommended Dataset Mix

For the first real training run, aim for roughly:

- 10-20% curated seed examples.
- 40-60% public benchmark-derived examples.
- 20-40% synthetic edge cases.

The exact mix can change. The important part is to keep a held-out evaluation set that is not purely generated by the same prompt used for training data.

## Week 2 Pipeline Target

The first `scripts/data_pipeline.py` should:

- load JSONL files from `data/examples/`,
- optionally load converted benchmark examples,
- validate required fields,
- deduplicate exact `(context, command, environment)` pairs,
- split into train/eval files,
- report label and risk-category counts.

## Labeling Rule

Always ask:

> Given this context and environment, is this command appropriate?

Do not ask:

> Does this command look scary?

That difference is what makes Sentinel valuable.
